# Model Sizing

只在拆分和排序 stories 时读取本文件。

## 默认策略

默认目标模型是 `sonnet`。只有用户显式传入 `--model opus` 时才放宽单 story 规模。

## Story 尺寸上限

| 模型 | code story 建议规模 | 复杂交互 | data-fill 批次 |
|------|---------------------|----------|----------------|
| `sonnet` | `<= 2-3` 个文件（含测试） | 每个 story 最多 1 个核心交互 | `<= 45` 条 |
| `opus` | `<= 5-6` 个文件（含测试） | 可以处理多组件协调，但仍要可验证 | `<= 60` 条 |

## 立即拆分信号

任一命中就拆：
- `modifies` 超过模型上限
- 一个 story 包含 2 个及以上独立用户交互
- 需要先重写复杂 mock 才能测试
- 同时包含“新增 i18n key”和“消费 i18n key”
- 同时跨越 schema / store / UI / 测试多个层级，且验收条件无法独立

## 保守原则

- 默认按最保守可验证方式拆分
- 宁可多几个小 story，也不要制造一个失败成本很高的大 story
- mock、schema、目录初始化、数据准备，优先拆成前置 story

## 排序原则

- 类型 / schema / data contract
- 工具函数 / 迁移准备 / mock 准备
- 组件或接口实现
- 交互逻辑
- 集成验证 / 汇总验证

多计划同时输入时：
- 被依赖计划优先
- 共享基础设施优先
- 验证型 story 收尾
